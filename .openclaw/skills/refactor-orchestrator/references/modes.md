# 工作模式说明

> 本文件用于帮助主控 agent 判断：这次任务应该以哪种方式进入 `refactor-orchestrator`。

## 总原则

先判断输入类型，再决定执行方式。

不要把所有任务都按同一种套路处理。

---

## Mode A：plan-driven

### 适用情况
当用户已经提供：
- 明确的重构计划文档
- PRD / architecture note / migration plan
- 已拆好的阶段或 checklist

### 主控 agent 应怎么做
1. 读取 governing doc
2. 初始化状态文件
3. 优先沿用原文档的 phase / milestone / task structure
4. 如原文档阶段不够清晰，再补充通用 phase model
5. 先做最小审计，再开始实现
6. 保持“一个最小任务 + 一个独立 QA”的循环

### 常见输入示例
- “按这个 plan 开始做”
- “根据这份重构方案推进”
- “按 PRD 执行”
- “根据 migration doc 继续”

### 注意事项
- 不要无故推翻原文档的阶段命名
- 不要一上来就扩 scope
- 文档中若有冲突或空白，先显式指出，再补充定义

---

## Mode B：audit-to-plan

### 适用情况
当用户只有目标，没有现成 plan。

例如：
- “帮我重构这个模块”
- “把这个脚本堆叠流程整理成正式链路”
- “这个系统太乱了，帮我拆一下”

### 主控 agent 应怎么做
1. 不直接分发大规模 implementation
2. 先分发 architect / pm 做现状审计
3. 输出最小 refactor brief / 阶段计划 / backlog
4. 明确项目约束与一票否决风险
5. 确认 phase 边界后，再进入最小实现任务

### 常见输入示例
- “帮我重构一下这个系统”
- “把这个流程正式化”
- “整理一下结构”

### 注意事项
- 没有 plan 时，不要假装已有 plan
- 审计输出不足以支撑实施时，先补定义，不要急着 coding

---

## Mode C：resume-from-status

### 适用情况
当项目已经做过一部分，用户说：
- “继续”
- “接着上次做”
- “从当前状态往下推进”

### 主控 agent 应怎么做
1. 读取状态文件
2. 查看最新 QA 结论
3. 查看最后一个被接受的任务
4. 确认当前 phase / gate
5. 只推进下一个最小、无 blocker 的任务
6. 如状态与代码现实不一致，先发起审计 / 核对

### 常见输入示例
- “继续”
- “接着做 P1”
- “从上次驳回点继续”

### 注意事项
- 不要凭印象恢复
- 不要跳过状态文件
- 不要把“已完成”误当成“已通过 QA”

---

## 模式选择优先级

当输入同时满足多个模式时，按以下优先级判断：

1. **resume-from-status**
   - 如果已有状态文件且用户明确要求继续，优先恢复，不重新开局

2. **plan-driven**
   - 如果有明确 governing doc，优先按文档推进

3. **audit-to-plan**
   - 当没有可靠计划文档时，先审计再产出计划

---

## 快速判定问题

主控 agent 在开始前，先问自己：

1. 用户有没有给现成计划文档？
2. 当前有没有状态文件？
3. 用户是在启动新任务，还是恢复旧任务？
4. 如果没有计划，是否足以直接实施，还是必须先审计？

如果以上问题答不清，不应直接进入 implementation。

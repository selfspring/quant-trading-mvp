## 项目概述

基于AI收集分析的多元市场量化交易系统，MVP阶段聚焦上期所黄金期货（AU），后续扩展至其他市场。

## 核心目标

- 30分钟K线波段交易 + 重大新闻事件快速反应
- 多信号融合决策（技术指标 + ML模型 + LLM新闻解读）
- **动态权重调整**（根据市场状态自适应）
- 完整风控体系
- 本地部署，Web监控界面

## 功能模块

### 1. 数据采集层

#### 1.1 行情数据
- **来源**: CTP实时行情接口（vnpy-ctp封装）
- **频率**: 30分钟K线（实时）
- **内容**: 开高低收、成交量、持仓量
- **存储**: TimescaleDB
- **质量监控**: ⚠️ **新增 data_quality_log 表**，记录采集成功率、延迟、错误信息

#### 1.2 基本面数据
- **来源**: 免费数据源（优先 AKShare，备用 TuShare）
- **内容**: 
  - 美联储利率决议
  - 非农就业数据
  - CPI通胀数据
  - 美元指数
  - 美债收益率
- **频率**: 定时爬取（每日/每周）
- **存储**: TimescaleDB
- **容错**: ⚠️ **新增多源备份**，AKShare 失效时自动切换 TuShare

#### 1.3 新闻数据
- **来源**: 
  - 新浪财经、东方财富（实时快讯）
  - 金十数据（可选，需谨慎使用）
  - ~~路透社、彭博RSS~~（已移除，数据源失效）
  - ~~Twitter/X~~（已移除，API成本高）
- **频率**: 实时监控
- **存储**: 
  - 原文: TimescaleDB（⚠️ **新增 content_hash 字段去重**）
  - Embedding: Chroma（用于相似度检索和LLM缓存）
- **去重机制**: ⚠️ **新增**，基于 content_hash 避免重复处理

**重要约束**: 
- 所有数据源必须免费
- AI负责发现数据源、生成采集脚本，但不直接爬取
- 需要注册的由用户操作

### 2. 信号生成层

#### 2.1 技术指标
- **库**: TA-Lib
- **指标**: 
  - 均线（MA）
  - MACD
  - RSI
  - 布林带（Bollinger Bands）
  - ATR（平均真实波幅，用于波动率过滤）
  - ⚠️ **扩展指标**：KDJ、CCI（存储在 extra_indicators JSONB 字段）
- **输入**: 30分钟K线
- **输出**: 实时计算的指标值

#### 2.2 ⚠️ 新增：市场状态识别
- **功能**: 识别当前市场状态（趋势市、震荡市、高波动市）
- **方法**: 基于 ADX 指标 + ATR 波动率
- **输出**: `'trending'`, `'ranging'`, `'volatile'`
- **用途**: 动态调整信号融合权重

#### 2.3 ML模型（LightGBM）
- **任务类型**: 回归（预测未来2根30分钟K线的收益率）
- **特征工程**: 
  - 价格数据（开高低收、成交量、持仓量）
  - 技术指标（MA、MACD、RSI等）
  - 基本面数据（美元指数、美债收益率等）
  - ⚠️ **新增市场状态特征**（ADX、ATR比率）
- **标签定义**: 未来2根K线（1小时后）的收益率
  - 公式: `(future_close - current_close) / current_close`
- **训练数据**: 历史数据（2020-2023年训练，2024年验证）
- **输出**: 
  - 预测收益率（连续值，如+0.8%或-0.5%）
  - 置信度（模型内部评估）
- **过拟合防范**: ⚠️ **新增 Walk-Forward 验证 + 正则化**

#### 2.4 LLM新闻解读（Claude Opus 4）
- **输入**: 实时抓取的新闻原文
- **输出**: 结构化JSON
  ```json
  {
    "importance": "high|medium|low|irrelevant",
    "direction": "bullish|bearish|neutral",
    "timeframe": "immediate|short-term|long-term",
    "confidence": 0.0-1.0,
    "reasoning": "简短解释"
  }
  ```
- **触发条件**: 
  - `importance >= medium` 且 `confidence >= 0.6` 时记录并可能触发交易
  - `importance == high` 且 `confidence >= 0.75` 时立即触发交易信号
- **性能优化**: ⚠️ **新增缓存机制**
  - Chroma 相似度检索（相似度 > 0.95 时复用历史分析）
  - Redis 缓存（24小时 TTL）
  - 预计节省 60% API 调用成本
- **超时降级**: ⚠️ **新增 5秒超时**，超时降级为 `neutral`

**重要约束**: 
- 不使用启发式规则平替AI判断
- 全部新闻都送给LLM判断，不做关键词预过滤

#### 2.5 多信号融合（已优化）
- **输入**: 
  - 技术指标信号
  - ML模型预测值+置信度
  - LLM新闻解读结果
  - ⚠️ **新增：市场状态**
- **融合逻辑**: 
  - ~~加权综合（权重可配置，初始建议: ML 50%, 技术指标 30%, LLM 20%）~~
  - ⚠️ **改为动态权重**：
    - 趋势市：技术指标 40%, ML 50%, LLM 10%
    - 震荡市：技术指标 50%, ML 40%, LLM 10%
    - 高波动市：技术指标 20%, ML 30%, LLM 50%
  - 方向一致性检查: 至少2/3的子信号方向一致才输出交易信号
- **输出**: 
  - 交易方向（买入/卖出/观望）
  - 信号强度（0.0-1.0，用于动态调仓）

### 3. 风控层

#### 3.1 信号过滤
- **波动率过滤**: 
  - 计算ATR，如果 `ATR > 近20日均值 * 2`，降低仓位50%或暂停交易
- **信号强度阈值**: 
  - ML模型置信度 < 0.65 时不交易
- **连败熔断**: 
  - 连续3笔交易亏损后，暂停交易1小时

#### 3.2 仓位管理
- **动态调仓**: 根据信号强度调整仓位
  - 信号强度 0.65-0.75: 30%仓位
  - 信号强度 0.75-0.85: 50%仓位
  - 信号强度 > 0.85: 70%仓位
- **最大仓位**: 不超过总资金的70%

#### 3.3 止损止盈
- **由ML模型给出**: 
  - 模型预测收益率的基础上，设定止损/止盈点位
  - 例如: 预测+0.8%，止盈设在+1.0%，止损设在-0.4%（盈亏比2:1）

#### 3.4 最大回撤控制
- **单周最大亏损**: 不超过总资金的25%
- **触发后**: 暂停交易，发送邮件告警

### 4. 交易执行层

#### 4.1 CTP接口（已明确）
- **功能**: 
  - 实时行情订阅（30分钟K线）
  - 下单（开仓/平仓）
  - 订单状态跟踪
  - 成交回报处理
- **封装库**: ⚠️ **明确使用 vnpy-ctp**（最成熟的Python封装）
- **模式**: 
  - MVP阶段使用模拟盘
  - 验证通过后切换真实盘
- **容错**: ⚠️ **新增心跳检测 + 自动重连**（最多3次）

#### 4.2 订单管理
- **订单类型**: 限价单（避免滑点过大）
- **订单状态**: 待提交、已提交、部分成交、全部成交、已撤单、失败
- **异常处理**: 
  - 下单失败重试（最多3次）
  - 超时未成交自动撤单

  - 2020-2023年训练
  - 2024年测试（验证是否过拟合）
- **性能优势**: ⚠️ **VectorBT 向量化计算，速度比 Backtrader 快 100 倍**

## 系统架构（重大变更）

### 架构模式：单进程多线程 → ⚠️ **多进程架构**

**变更理由**：Python GIL 限制，多线程无法利用多核

**新架构**：
```
进程1：数据采集（CTP + 新闻爬虫 + 基本面）
进程2：信号生成（技术指标 + ML + LLM + 融合）
进程3：风控 + 交易执行（信号过滤 + 仓位管理 + CTP下单）
进程4：Web服务（FastAPI + Vue + WebSocket）
进程5：监控告警（邮件 + 日志 + 健康检查）

进程间通信：Redis Pub/Sub
进程守护：Supervisor（自动重启）
```

## 关键约束

1. **AI不降级**: 在需求文档标注"AI"的位置，不使用启发式规则平替
2. **免费数据源**: 所有数据源必须免费，需要注册的由用户操作
3. **本地部署**: 系统运行在本地，不依赖云服务（除了LLM API）
4. **模拟盘先行**: 真实交易前必须通过模拟盘验证
5. ⚠️ **新增：多进程架构**: 必须使用多进程，不能用多线程
6. ⚠️ **新增：进程守护**: 必须使用 Supervisor 管理进程
7. ⚠️ **新增：数据去重**: 新闻数据必须去重
8. ⚠️ **新增：LLM缓存**: 必须实现 LLM 结果缓存

## 故障处理与降级方案（新增章节）

### 数据源失效

| 场景 | 检测方式 | 降级策略 | 恢复策略 |
|-----|---------|---------|---------|
| CTP 断线 | 心跳超时 | 暂停交易，发送告警 | 自动重连（最多3次） |
| 新闻爬虫失败 | 连续3次失败 | 切换备用源（东方财富 → 新浪） | 每5分钟重试 |
| AKShare 失效 | HTTP 错误 | 切换 TuShare | 每小时重试 |
| LLM API 限流 | 429 错误 | 降级为 neutral，信号权重降低 | 指数退避重试 |

### 模型失效

| 场景 | 检测方式 | 降级策略 |
|-----|---------|---------|
| ML 模型预测异常 | 置信度 < 0.3 连续10次 | 降低 ML 权重至 20% |
| LLM 解读质量差 | 人工标注准确率 < 60% | 降低 LLM 权重至 10% |
| 技术指标计算错误 | 数值超出合理范围 | 跳过该K线，使用上一根 |

### 系统异常

| 场景 | 检测方式 | 应对措施 |
|-----|---------|---------|
| 进程崩溃 | Supervisor 监控 | 自动重启 + 邮件告警 |
| Redis 内存溢出 | 内存使用率 > 90% | 清理过期缓存 + 告警 |
| 数据库连接失败 | 连接超时 | 重试3次 + 告警 |

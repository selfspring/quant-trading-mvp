### 1.2 技术选型变更理由

#### 变更1：Backtrader → VectorBT
**问题**：Backtrader 是单线程框架，处理大量数据时性能差  
**解决**：VectorBT 基于 NumPy 向量化计算，速度快 100 倍  
**影响**：回测模块需要重新设计 API

#### 变更2：CTP → vnpy-ctp
**问题**：原文档只写"CTP"，没明确用哪个 Python 封装库  
**解决**：明确使用 vnpy 的 CTP 封装（最成熟，社区活跃）  
**备选**：openctp-ctp（官方维护，但文档较少）

#### 变更3：新增 Redis
**问题**：多进程架构需要消息队列  
**解决**：Redis 用于进程间通信、LLM 结果缓存、实时数据缓存  
**可选**：MVP 阶段可用 multiprocessing.Queue 替代

#### 变更4：新增 Supervisor
**问题**：进程崩溃后无法自动重启  
**解决**：Supervisor 管理所有子进程，自动重启 + 日志管理

### 4.1 ⚠️ 新增模块：市场状态识别器

```python
class MarketRegimeDetector:
    """
    识别市场状态：趋势市、震荡市、高波动市
    用于动态调整信号融合权重
    """
    def detect(self, klines: List[KlineData]) -> str:
        """
        返回：'trending', 'ranging', 'volatile'
        """
        # 方法1：基于ADX指标
        adx = self.calculate_adx(klines)
        if adx > 25:
            return 'trending'
        elif adx < 20:
            return 'ranging'
        
        # 方法2：基于ATR波动率
        atr_ratio = self.calculate_atr_ratio(klines)
        if atr_ratio > 1.5:
            return 'volatile'
        
        return 'ranging'
```

**新增理由**：不同市场状态应该用不同策略，固定权重不合理

### 4.2 🔄 优化模块：信号融合器（动态权重）

```python
class AdaptiveSignalFusion:
    """
    根据市场状态动态调整信号融合权重
    """
    def fuse_signals(
        self, 
        technical: TechnicalIndicators,
        ml_pred: Prediction,
        news_analysis: List[NewsAnalysis],
        market_regime: str  # ⚠️ 新增参数
    ) -> TradingSignal:
        
        # 根据市场状态调整权重
        if market_regime == 'trending':
            weights = {'technical': 0.4, 'ml': 0.5, 'news': 0.1}
        elif market_regime == 'volatile':
            weights = {'technical': 0.2, 'ml': 0.3, 'news': 0.5}
        else:  # ranging
            weights = {'technical': 0.5, 'ml': 0.4, 'news': 0.1}
        
        # 加权融合
        technical_score = self._calc_technical_score(technical)
        ml_score = ml_pred.confidence * (1 if ml_pred.predicted_return > 0 else -1)
        news_score = self._calc_news_score(news_analysis)
        
        final_score = (
            weights['technical'] * technical_score +
            weights['ml'] * ml_score +
            weights['news'] * news_score
        )
        
        # 方向一致性检查
        signals = [technical_score, ml_score, news_score]
        positive_count = sum(1 for s in signals if s > 0)
        
        if positive_count < 2:  # 至少2/3一致
            return TradingSignal(signal_type='hold', signal_strength=0)
        
        return TradingSignal(
            signal_type='buy' if final_score > 0 else 'sell',
            signal_strength=abs(final_score)
        )
```

### 4.3 🔄 优化模块：LLM新闻解读器（增加缓存）

```python
class LLMAnalyzer:
    def __init__(self, redis_client, chroma_client):
        self.redis = redis_client
        self.chroma = chroma_client
        self.cache_ttl = 3600 * 24  # 缓存24小时
    
    async def analyze_news(self, news: NewsItem) -> NewsAnalysis:
        """
        优化策略：
        1. 先查 Redis 缓存（快速）
        2. 再查 Chroma 相似新闻（中速）
        3. 最后调用 Claude API（慢速）
        """
        # 1. Redis 缓存查询
        cache_key = f"news_analysis:{news.content_hash}"
        cached = self.redis.get(cache_key)
        if cached:
            return NewsAnalysis.from_json(cached)
        
        # 2. Chroma 相似度检索
        similar_news = self.chroma.query(
            query_embeddings=[news.embedding],
            n_results=3
        )
        
        if similar_news and similar_news['distances'][0][0] < 0.05:  # 相似度 > 0.95
            # 复用历史分析结果
            similar_id = similar_news['ids'][0][0]
            analysis = self.get_cached_analysis(similar_id)
            
            # 写入 Redis 缓存
            self.redis.setex(cache_key, self.cache_ttl, analysis.to_json())
            return analysis
        
        # 3. 调用 Claude API（设置超时）
        try:
            analysis = await asyncio.wait_for(
                self.call_claude_api(news),
                timeout=5.0  # 5秒超时
            )
        except asyncio.TimeoutError:
            # 超时降级为 neutral
            analysis = NewsAnalysis(
                importance='low',
                direction='neutral',
                confidence=0.5,
                reasoning='LLM timeout, fallback to neutral'
            )
        
        # 写入缓存
        self.redis.setex(cache_key, self.cache_ttl, analysis.to_json())
        return analysis
```

**优化效果**：
- 缓存命中率预计 60-70%
- API 调用量减少 60%
- 成本降低 60%（$15/1M tokens → $6/1M tokens）

## 六、故障处理与降级方案（新增章节）

### 6.1 数据源失效

| 场景 | 检测方式 | 降级策略 | 恢复策略 |
|-----|---------|---------|---------|
| CTP 断线 | 心跳超时 | 暂停交易，发送告警 | 自动重连（最多3次） |
| 新闻爬虫失败 | 连续3次失败 | 切换备用源（东方财富 → 新浪） | 每5分钟重试 |
| AKShare 失效 | HTTP 错误 | 切换 TuShare | 每小时重试 |
| LLM API 限流 | 429 错误 | 降级为 neutral，信号权重降低 | 指数退避重试 |

### 6.2 模型失效

| 场景 | 检测方式 | 降级策略 |
|-----|---------|---------|
| ML 模型预测异常 | 置信度 < 0.3 连续10次 | 降低 ML 权重至 20% |
| LLM 解读质量差 | 人工标注准确率 < 60% | 降低 LLM 权重至 10% |
| 技术指标计算错误 | 数值超出合理范围 | 跳过该K线，使用上一根 |

> **版本说明**：基于 PM 评审意见完善，调整时间预估、优先级，增加分阶段交付计划和成本控制方案。

## 项目概述

基于AI收集分析的多元市场量化交易系统，MVP阶段聚焦上期所黄金期货（AU），后续扩展至其他市场。

## 核心目标

- 30分钟K线波段交易 + 重大新闻事件快速反应
- 多信号融合决策（技术指标 + ML模型 + LLM新闻解读）
- **动态权重调整**（根据市场状态自适应）
- 完整风控体系 + 故障降级方案
- 本地部署，Web监控界面
- **消息流追踪**（全链路可观测）

## 功能模块

### 1. 数据采集层

#### 1.1 行情数据
- **来源**: CTP实时行情接口（openctp-ctp封装）
- **频率**: 30分钟K线（实时）
- **内容**: 开高低收、成交量、持仓量
- **存储**: PostgreSQL（原设计 TimescaleDB，实际使用普通 PostgreSQL，未启用超表）
- **质量监控**: data_quality_log 表，记录采集成功率、延迟、错误信息
- **消息追踪**: 每条K线数据带 trace_id

> **实际实现变更（2026-03-13 ~ 2026-03-16）**：
> - SimNow 日盘不推送实时 Tick，实现**三级数据回退架构**：
>   - Level 1: SimNow 实时 Tick（优先，夜盘可用）
>   - Level 2: 天勤量化（TqSdk）历史 K 线（30分钟线 10000 根，2023-12 ~ 2026-03）
>   - Level 3: AkShare 日线兜底
> - 采集进程（`run_ctp_collector.py`）长驻运行，策略脚本（`run_single_cycle.py`）每5分钟单次触发
> - K 线时间戳去重：策略脚本检测 `last_kline_time`，30分钟线未更新时直接跳过预测

#### 1.2 基本面数据
- **来源**: 优先 AKShare，备用 TuShare
- **内容**: 美联储利率、非农、CPI、美元指数、美债收益率
- **频率**: 定时爬取（每日/每周）
- **存储**: PostgreSQL
- **容错**: 多源备份，AKShare 失效时自动切换

> **实际实现（2026-03-16）**：⚠️ **暂未实现**，MVP 阶段基本面数据未接入，当前仅使用技术指标 + ML 信号

#### 1.3 新闻数据
- **来源**: 新浪财经、东方财富
- **频率**: 实时监控
- **存储**: 
  - 原文: TimescaleDB（含 content_hash 去重）
  - Embedding: Chroma（用于相似度检索和LLM缓存）
- **去重机制**: 基于 content_hash 唯一索引

> **实际实现（2026-03-16）**：⚠️ **暂未实现**，MVP 阶段新闻数据未接入，LLM 新闻解读模块未开发

### 2. 信号生成层

#### 2.1 技术指标

**原需求**：
- 均线系统（MA5/MA10/MA20/MA60）
- MACD、RSI、布林带
- ATR（波动率）

**实现方案**（更新于 2026-03-11）：
- **模块位置**：`quant/signal_generator/technical_indicators.py`
- **设计原则**：纯函数，输入 DataFrame，输出带指标的 DataFrame
- **复用性**：被 ML 特征工程、技术分析信号、回测系统共同使用
- **依赖**：pandas、numpy（考虑使用 pandas_ta 或 ta-lib）
- **库**: TA-Lib
- **指标**: MA、MACD、RSI、布林带、ATR
- **扩展指标**: KDJ、CCI（存储在 extra_indicators JSONB 字段）
- **输入**: 30分钟K线
- **输出**: 实时计算的指标值

#### 2.2 市场状态识别（新增）
- **功能**: 识别趋势市、震荡市、高波动市
- **方法**: ADX 指标 + ATR 波动率
- **输出**: `'trending'`, `'ranging'`, `'volatile'`
- **用途**: 动态调整信号融合权重

#### 2.3 ML模型（LightGBM）
- **任务类型**: 回归（预测未来1小时收益率）
- **特征工程**: 价格 + 技术指标 + 基本面 + 市场状态
- **标签定义**: `(future_close - current_close) / current_close`
- **训练数据**: 2020-2023年训练，2024年验证
- **输出**: 预测收益率 + 置信度
- **过拟合防范**: Walk-Forward 验证 + 正则化

> **实际实现变更（2026-03-16）**：
> - **特征数量**: 18 → **47 个**（新增动量、波动率、成交量、持仓量、均线交叉、价格形态、时间特征）
> - **训练数据**: 模拟数据 500 根 → **天勤量化真实 30 分钟线 10000 根**（2023-12 ~ 2026-03）
> - **方向准确率**: 44.93% → **64.01%**，相关系数: -0.093 → **0.2109**
> - **置信度公式修正**: 异常预测值（>5%）自动降低置信度，防止极端发单
> - **置信度阈值**: 0.65（低于此值不交易）
> - 基本面特征暂未接入，`open_oi`/`close_oi` 字段已从训练特征中排除（实时数据无此字段）

#### 2.4 LLM新闻解读（Claude Opus 4）
- **输入**: 实时抓取的新闻原文
- **输出**: 结构化JSON（importance, direction, timeframe, confidence, reasoning）
- **触发条件**: 
  - `importance >= medium` 且 `confidence >= 0.6` 时记录
  - `importance == high` 且 `confidence >= 0.75` 时立即触发交易信号
- **性能优化**: 
  - Chroma 相似度检索（相似度 > 0.95 时复用）

> **实际实现（2026-03-16）**：⚠️ **暂未实现**，MVP 阶段 LLM 新闻解读未开发
  - Redis 缓存（动态 TTL：2-24小时）
  - 预计节省 60% API 成本
- **超时降级**: 5秒超时，降级为 `neutral`
- **缓存失效**: 重大事件（美联储决议、非农等）触发主动清理

#### 2.5 多信号融合（动态权重）
- **输入**: 技术指标 + ML预测 + LLM解读 + 市场状态
- **融合逻辑**: 
  - 趋势市：技术指标 40%, ML 50%, LLM 10%
  - 震荡市：技术指标 50%, ML 40%, LLM 10%
  - 高波动市：技术指标 20%, ML 30%, LLM 50%
  - 方向一致性检查: 至少2/3一致
- **输出**: 交易方向 + 信号强度（0.0-1.0）

> **实际实现（2026-03-16）**：⚠️ **部分实现**，LLM 信号未接入，当前仅融合技术指标 + ML 预测两路信号，动态权重逻辑简化为 ML 置信度直接作为信号强度

### 3. 风控层

#### 3.1 信号过滤
- **波动率过滤**: `ATR > 近20日均值 * 2` 时降低仓位50%
- **信号强度阈值**: ML置信度 < 0.65 时不交易
- **连败熔断**: 连续3笔亏损后暂停1小时

#### 3.2 仓位管理
- **动态调仓**: 
  - 信号强度 0.65-0.75: 30%仓位
  - 信号强度 0.75-0.85: 50%仓位
  - 信号强度 > 0.85: 70%仓位
- **最大仓位**: 70%

#### 3.3 止损止盈
- **由ML模型给出**: 盈亏比 2:1
- **示例**: 预测+0.8%，止盈+1.0%，止损-0.4%

#### 3.4 最大回撤控制
- **单周最大亏损**: 25%
- **触发后**: 暂停交易 + 邮件告警

### 4. 交易执行层

#### 4.1 CTP接口

**原设计**：vnpy-ctp 6.6.9+

**实际使用**（变更于 2026-03-11）：
- **选型**: openctp-ctp
- **变更原因**: API 完全兼容，功能完整，社区活跃
- **影响评估**: 无影响，接口调用方式一致

**功能**: 实时行情订阅、下单、订单跟踪、成交回报
- **模式**: MVP阶段使用模拟盘
- **容错**: 心跳检测 + 自动重连（最多3次）
- **消息追踪**: 每个订单带 trace_id

#### 4.2 订单管理
- **订单类型**: 限价单
- **订单状态**: 待提交、已提交、部分成交、全部成交、已撤单、失败
- **异常处理**: 下单失败重试（最多3次）+ 超时撤单

> **实际实现变更（2026-03-18）**：
> - 限价单价格策略：买入用当前价 +5跳（+0.10元），卖出用当前价 -5跳，保证 SimNow 能成交
> - 原市价单（price=0）在 SimNow 会被立即撤单（ErrorID=5），已修复
> - 订单状态回调通过 `OnRtnOrder`/`OnRtnTrade` 记录日志，暂无持久化到数据库
> - 重试逻辑暂未实现，每次策略触发最多发1笔单（冷却期 300 秒）

## 系统架构（多进程设计）

### 架构模式：多进程 + 消息追踪

```
进程1：数据采集（CTP + 新闻 + 基本面）
进程2：信号生成（技术指标 + ML + LLM + 融合）
进程3：风控 + 交易执行（信号过滤 + 仓位管理 + CTP下单）
进程4：Web服务（FastAPI + Vue + WebSocket）
进程5：监控告警（邮件 + 日志 + 健康检查）

进程间通信：Redis Pub/Sub + 消息追踪（trace_id）
进程守护：Supervisor（自动重启）或本地脚本
```

> **实际实现架构（2026-03-16）**：
> ```
> 长驻进程：run_ctp_collector.py（每交易时段启动一次）
>   └─ SimNow 实时 Tick → 聚合 1 分钟 K 线 → PostgreSQL
>
> 定时触发：run_single_cycle.py（Windows Task Scheduler，每 5 分钟）
>   └─ 读 30 分钟 K 线（数据库 → 天勤 → AkShare 三级回退）
>   └─ K 线时间戳去重（30 分钟内重复触发直接跳过）
>   └─ 47 特征工程 → LightGBM 预测 → 置信度过滤（>0.65）
>   └─ 信号处理（反向信号先平仓）→ 风控检查
>   └─ 按需连接 CTP 交易接口 → 限价单发单（当前价±5跳）
>   └─ 持仓/状态持久化到 strategy_state.json
>
> 进程间通信：strategy_state.json 文件（替代 Redis Pub/Sub）
> 进程守护：Windows Task Scheduler（替代 Supervisor）
> 未实现：进程2（LLM）、进程4（Web）、进程5（告警）
> ```

## 成本控制方案（新增）

### LLM API 成本控制

| 措施 | 预期效果 | 实施优先级 |
|-----|---------|-----------|
| **Chroma 相似度检索** | 节省 40% | P1 |
| **Redis 缓存（动态 TTL）** | 节省 20% | P1 |
| **每日调用上限（500次）** | 避免失控 | P1 |
| **缓存预热（开盘前）** | 减少实时调用 | P2 |
| **批量分析（非交易时段）** | 降低峰值成本 | P2 |

**预期成本**：
- 无缓存：$50-100/月

## 故障处理与降级方案

### 数据源失效

| 场景 | 检测方式 | 降级策略 | 恢复策略 |
|-----|---------|---------|---------|

| 场景 | 检测方式 | 降级策略 |
|-----|---------|---------|

| 技术指标错误 | 数值超出范围 | 跳过该K线 |

### 系统异常

| 场景 | 检测方式 | 应对措施 |

| 数据库连接失败 | 连接超时 | 重试3次 + 告警 |

### 3. 发现的问题

#### 严重问题（阻塞发布）

1. **无单元测试框架**
   - 位置：整个项目
   - 影响：无法自动化验证代码正确性，回归测试成本高
   - 复现步骤：运行 `python -m pytest --version` 失败
   - 建议修复：安装 pytest，添加 `requirements-dev.txt`

2. **tests/ 目录不存在**
   - 位置：项目根目录
   - 影响：测试代码无处存放，不符合 Python 项目规范
   - 建议修复：创建 `tests/` 目录，按模块组织测试文件

3. **核心模块无测试**
   - 位置：`quant/common/config.py`, `quant/common/tracer.py`, `quant/common/db_pool.py`
   - 影响：配置验证、消息追踪、数据库连接池等核心逻辑无测试保障
   - 建议修复：为每个模块添加单元测试，覆盖正常和异常场景

#### 一般问题（需要修复）

4. **测试脚本无自动化断言**
   - 位置：`scripts/test_*.py`
   - 影响：无法判断测试是否通过，依赖人工检查日志
   - 建议修复：使用 `assert` 语句或 pytest 断言

5. **错误处理不完善 - tracer.py**
   - 位置：`quant/common/tracer.py:156-170`
   - 影响：队列满时非关键事件被丢弃但无告警
   - 建议修复：添加丢弃事件的定期告警机制

6. **错误处理不完善 - ctp_market.py**
   - 位置：`quant/data_collector/ctp_market.py:155-165`
   - 影响：CTP 连接超时仅抛出异常，无重试机制
   - 建议修复：实现指数退避重试（最多 3 次）

7. **日志记录不充分**
   - 位置：多个文件
   - 影响：调试困难，无法追踪问题根因
   - 建议修复：
     - 增加关键路径的 DEBUG 日志
     - 统一日志格式和级别
     - 添加日志轮转配置

8. **配置验证不足**
   - 位置：`quant/common/config.py`
   - 影响：配置错误可能在运行时才暴露
   - 建议修复：
     - 增加配置项范围验证（如权重和=1.0）
     - 添加配置加载后的自检函数

9. **依赖管理问题**
   - 位置：`requirements.txt`
   - 影响：`vnpy` 模块未安装但代码中有引用
   - 复现：运行 `test_ctp_module.py` 报告 VN.PY 导入失败
   - 建议修复：更新 requirements.txt 或移除未使用依赖

10. **无 CI/CD 配置**
    - 位置：项目根目录
    - 影响：代码变更无法自动触发测试
    - 建议修复：添加 `.github/workflows/ci.yml` 或类似配置

#### 轻微问题（可延后）

11. **测试脚本命名不规范**
    - 位置：`scripts/test_*.py`
    - 影响：与 Python 测试惯例不符
    - 建议修复：移动到 `tests/` 目录，命名为 `test_*.py`

12. **文档中的测试要求不明确**
    - 位置：`PRD-v3.md` 交付标准
    - 影响：测试覆盖目标模糊
    - 建议修复：明确测试覆盖率目标（如 >80%）

# LLM 新闻链路重构项目约束

## 1. 项目基本信息
- 项目 / 模块 / 子系统名称：LLM 新闻链路重构
- governing doc / brief：`docs/LLM_NEWS_PIPELINE_REFACTOR_PLAN.md`
- 当前工作模式：plan-driven
- 当前阶段：P0

---

## 2. 关键语义约束

- 约束 1：
  - 对象：`news_raw`
  - 必须表达：原始新闻采集结果
  - 不得混入：候选筛选、方向判断、验证标签
  - 原因：保持原始数据层语义单一，避免后续研究口径污染

- 约束 2：
  - 对象：`news_candidates`
  - 必须表达：是否值得进入下一步分析
  - 不得混入：最终 truth label、方向结论、验证结果
  - 原因：候选筛选应是轻语义，不应承担最终判断

- 约束 3：
  - 对象：`news_analysis`
  - 必须表达：LLM 分析结果
  - 不得混入：价格验证结果、回测标签、未来结果
  - 原因：分析层与验证层必须解耦

- 约束 4：
  - 对象：时间字段
  - 必须表达：`published_time`、`analyzed_at`、`effective_time` 各自唯一语义
  - 不得混入：用单一 `time` 字段混合承载多重核心语义
  - 原因：时间语义稳定是验证口径可信的前提

---

## 3. 一票否决风险

- 风险 1：
  - 描述：future labels 或验证结果回灌到实时分析 prompt
  - 触发信号：prompt 构造路径中直接或间接使用未来价格变化/correct 标签
  - 为什么是一票否决：会直接破坏“实时预测能力”的可信性

- 风险 2：
  - 描述：`published_time / analyzed_at / effective_time` 继续混用
  - 触发信号：代码、表结构、验证逻辑仍使用含混 `time` 承担多重语义
  - 为什么是一票否决：会导致验证与回测结论不可解释

- 风险 3：
  - 描述：analysis 与 verification 继续混层
  - 触发信号：验证结果继续回写到 `news_analysis` 主语义或实时分析路径
  - 为什么是一票否决：会污染分析定义，导致研究问题不再可分辨

---

## 4. 当前阶段明确不做
本阶段禁止顺手扩展到以下内容：

- RAG 增强实验
- signal fusion 接回
- strategy backtest 优化
- 以收益结果替代方向/筛选问题验证

---

## 5. 验证口径
- 本阶段验证目标：先建立 P0 定义清理与主入口统一基础
- 验证对象：主入口、职责边界、时间字段语义、candidate screening 定位
- 验证方法：代码审计、文件读回、调用链检查、独立 QA
- 验证环境：仓库当前代码与文档
- 验证输入范围：LLM 新闻分析相关脚本、模块、表语义
- 通过标准：P0 边界清晰、最小任务顺序明确、无语义偷换
- 明确不作为本轮验证依据的内容：策略收益、RAG 提升效果、回测结果

---

## 6. 兼容性 / 迁移要求
- 需要兼容的对象：现有脚本调用路径、现有数据表语义（在明确迁移前不得静默破坏）
- 可接受的破坏范围：P0 内允许明确废弃重复入口，但需说明替代路径
- 不可破坏的行为：不得在未定义迁移路径前让主链路不可恢复
- 是否需要迁移脚本：后续可能需要
- 是否需要回滚方案：如涉及入口切换和 schema 调整，需要
- 是否需要双写 / 双读 / 灰度：P0 暂不要求，P1/P2 视改造范围决定

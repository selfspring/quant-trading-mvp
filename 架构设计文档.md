# 量化交易系统 MVP - 架构设计文档

## 一、技术选型评审

### 1.1 核心技术栈评审

| 技术 | PM建议 | 架构评审 | 最终决策 |
|------|--------|---------|---------|
| **开发语言** | Python 3.10+ | ✅ 合理。量化交易生态成熟，库丰富 | Python 3.10+ |
| **时序数据库** | TimescaleDB | ✅ 合理。PostgreSQL扩展，SQL友好，压缩率高 | TimescaleDB 2.x |
| **向量数据库** | Chroma | ✅ 合理。轻量级，本地部署，适合MVP | Chroma 0.4+ |
| **机器学习** | LightGBM | ✅ 合理。训练快，效果好，适合表格数据 | LightGBM 4.x |
| **回测框架** | Backtrader | ✅ 合理。成熟稳定，文档完善 | Backtrader 1.9+ |
| **交易接口** | CTP | ✅ 必选。上期所官方接口 | CTP (openctp) |
| **AI模型** | Claude Opus 4 | ✅ 合理。新闻理解能力强 | Claude Opus 4 API |
| **前端** | React/Vue | 🔄 建议简化为 Vue 3 + Vite | Vue 3 + Vite |
| **消息队列** | 未提及 | ⚠️ 建议增加 Redis | Redis 7.x (可选) |

### 1.2 新增建议

1. **Redis**：用于模块间异步通信和缓存
   - 信号生成 → 风控 → 交易执行的消息传递
   - 实时数据缓存（最新K线、指标值）
   - 可选：MVP阶段可用内存队列替代

2. **日志系统**：structlog + 文件轮转
   - 结构化日志便于问题排查
   - 按模块分文件存储

3. **配置管理**：pydantic + YAML
   - 类型安全的配置验证
   - 环境变量覆盖支持

   CTP → market_collector → TimescaleDB

## 六、关键设计决策

### 决策1：数据库选型 - TimescaleDB vs InfluxDB
**选择**：TimescaleDB  
**理由**：
- SQL友好，开发效率高
- 支持复杂查询和JOIN
- 压缩率高，存储成本低
- 与PostgreSQL生态兼容

### 决策2：模块间通信 - 同步 vs 异步
**选择**：混合模式  
**理由**：
- 数据采集 → 信号生成：异步（通过数据库）
- 信号生成 → 风控 → 交易：同步（保证实时性）
- MVP阶段可用内存队列，后续可升级为Redis

### 决策3：前端技术 - React vs Vue
**选择**：Vue 3 + Vite  
**理由**：
- 学习曲线平缓
- 开发效率高
- Vite构建速度快
- 适合单人开发

### 决策4：部署方式 - 单机 vs 分布式
**选择**：单机部署  
**理由**：
- MVP阶段数据量小
- 降低运维复杂度
- 后续可平滑升级为分布式

### 决策5：ML模型部署 - 在线训练 vs 离线训练
**选择**：离线训练 + 在线推理  
**理由**：
- 训练耗时长，不适合实时
- 定期重训练（每周/每月）
- 推理速度快（<100ms）

## 七、技术风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| CTP接口不稳定 | 高 | 增加重连机制 + 心跳检测 |
| 数据源失效 | 中 | 准备多个备用数据源 |
| ML模型效果差 | 高 | 预留特征工程和调参时间 |
| LLM API限流 | 中 | 增加重试机制 + 降级策略 |
| 数据库性能瓶颈 | 低 | TimescaleDB压缩 + 索引优化 |
| 前端开发延期 | 低 | 先用命令行界面，后续迭代 |

# 通用阶段模型

> 本文件定义 `refactor-orchestrator` 默认使用的阶段模型。
> 如果 governing doc 已经有清晰的 phase / milestone / iteration 命名，优先沿用原文档。

## 总原则

- 优先沿用 governing doc 的阶段命名
- 如果原文档没有清晰阶段，再使用本文件的默认阶段模型
- 阶段的意义是防止“边做边漂移”，不是为了制造流程负担

---

## Phase 0：现状审计

### 目标
- 搞清当前系统/模块/链路如何工作
- 找到隐藏依赖、重复入口、职责重叠、语义混乱点
- 输出当前风险与最小推进顺序

### 常见产物
- 当前结构说明
- 调用链 / 数据流 / 模块职责映射
- 当前问题清单
- 风险清单
- 最小推进建议

### 未完成前不应做
- 大规模实现
- 扩 scope 优化
- 大面积接口/字段重定义

---

## Phase 1：目标定义

### 目标
- 明确目标结构、边界、契约、语义定义
- 明确当前阶段 in-scope / out-of-scope
- 明确项目约束与一票否决风险

### 常见产物
- 目标结构说明
- schema / interface / module boundary 定义
- 项目约束清单
- 阶段边界说明

### 未完成前不应做
- 多任务并行改造
- 边实现边重新定义目标
- 把验证、优化、增强混在一起

---

## Phase 2：最小重构执行

### 目标
- 一次执行一个最小任务
- 保持任务边界稳定
- 每轮都要有可验证产物

### 常见产物
- 小范围代码改动
- 小范围结构调整
- 最小迁移步骤
- 与本轮目标对应的验证结果

### 执行规则
- 一个最小任务一个 agent
- backend 完成后必须独立 QA
- 没通过 QA，不进入下一任务

---

## Phase 3：验证与收尾

### 目标
- 独立 QA
- 文档同步
- migration notes / runbook / status / work-log 完整

### 常见产物
- QA 结论
- 被接受 / 被驳回记录
- 文档更新
- 工作日志
- 下一步建议

### 未完成前不应做
- 宣告阶段完成
- 跳到下一阶段
- 忽略驳回项

---

## Phase 4：增强与后续优化（可选）

### 目标
- 在基础重构稳定后，再做增强、优化、扩展

### 常见内容
- 性能优化
- 体验优化
- 扩展能力
- 清理遗留技术债

### 前提
- 前面阶段已经稳定
- 基础结构和语义没有悬而未决的问题
- 当前增强不会破坏已建立边界

---

## 阶段切换规则

只有满足以下条件，才应进入下一阶段：

1. 当前阶段目标已清晰交付
2. 已有独立 QA 或等价验证
3. 状态文件已更新
4. 当前 blocker 已被处理或显式记录
5. 下一阶段的边界已明确

如果这些条件不满足，不应因为“代码能跑”就切阶段。

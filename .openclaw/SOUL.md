## 工作流程

### 1. 接收需求
倾听 → 提出澄清问题 → 确认理解正确

### 2. 任务规划
拆分成小任务 → 评估难度 → 确定优先级 → 选择合适 agent

### 3. 任务分发
- 照 `docs/task-template.md` 模板编写任务说明
- 提供上下文：环境信息、相关文件、已知坑
- **任务末尾必须附加强制验证清单**（读回文件、跑测试、逐条对照需求、汇报结果）
- 用 `sessions_spawn` 分配

### 4. 进度跟踪
等待 completion event（不主动轮询）→ 检查结果质量 → 发现问题及时调整

### 4.5 QA 验收（强制）
开发 agent 完成后，**必须**再 spawn 一个 QA agent 独立验证：
- 不能让开发 agent 自测
- QA agent 照 `docs/task-template.md` 的 QA 模板编写任务
- QA 通过 → 进入汇报；QA 失败 → 分析问题 → 重新分发给开发 agent

### 5. 结果汇报 + 工作日志（强制）
整理成果 → 简洁汇报 → 指出关键点 → 提供下一步建议

**每次任务结束后必须写工作日志**，不等用户提醒：
- 路径：`E:\quant-trading-mvp\docs\work-log\YYYY-MM-DD-HH-任务名.md`
- 内容：已完成的步骤 / 发现的问题 / 遗留事项 / 下一步
- 目的：上下文清除后可快速恢复现场

### 6. 经验记录
被纠正或任务失败后，立即写入 `~/self-improving/corrections.md` 和 `memory.md`

| Agent | 职责 |

| tech-writer | 技术文档 |
| pm | 产品经理（需求分析、PRD）|
| code-artisan | 代码工匠（优化、重构）|

**选 agent 原则**：一个任务一个 agent，任务要小而具体，提供清晰的输入和期望输出。

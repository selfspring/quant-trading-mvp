# Runtime Note - au9999 1m 数据覆盖审计（2013-01-02 邻近窗口）

## 摘要
本轮只做审计，不修数据、不改表、不改验证逻辑。目标是确认 `au9999` 在 `2013-01-02 18:33:56+08` 邻近时间段的 `1m` 数据缺口分布，并判断这是局部缺失还是更系统性的历史覆盖问题。

结论先行：
- **`au9999` 在 2012-12-31 ~ 2013-01-04 之间存在明显的跨多天 1m 大缺口。**
- 关键大缺口为：`2012-12-31 15:00:00+08 -> 2013-01-04 09:01:00+08`，约 **90.017 小时**。
- 但从更宽范围看，`2012-12`、`2013-01` 的月度 / 周度覆盖总体是连续且规则的，只有 `2012-12-31` 所在周显著异常。
- **当前更稳妥的判断是：这是 `au9999` 的“局部时间段集中缺失”，而非已被证实的整段历史系统性缺失。**
- 是否属于“全局历史库问题”证据不足，因为同窗口内其他 `au*` symbol 几乎没有 1m 数据，无法形成有效横向对照。

## 审计方式摘要
因 PowerShell 下内联 SQL 转义较容易踩坑，本轮改用临时 Python + `psycopg2` 执行只读查询。

主要查询分为 4 类：

2. **关键窗口最大连续缺口**：对窗口内 `1m` 时间序列做 `LEAD()`，按相邻记录间隔降序取 Top N
3. **扩展范围覆盖**：统计 `2012-11 ~ 2013-02` 的月度覆盖、`2012-12 ~ 2013-01` 的周度覆盖
4. **有限对照**：查看同窗口 `au9999` 之外若干 `au*` symbol 的 `1m` 是否存在，以判断能否做横向比较

### 1.2 关键观察
- 该窗口内实际只有 **2 个交易日**有 `1m` 数据：
  - `2012-12-31`
  - `2013-01-04`
- 这两天都只有 **225 条**，时间范围都为 **09:01 ~ 15:00**。
- 已知 anchor `2013-01-02 18:33:56+08` 落在完全无 `1m` 数据的空白区间中。

### 1.3 最大连续缺口
窗口内 Top gaps：
1. `2012-12-31 15:00:00 -> 2013-01-04 09:01:00` = **90.017 小时**
2. `2012-12-31 11:30:00 -> 2012-12-31 13:31:00` = **2.017 小时**
3. `2013-01-04 11:30:00 -> 2013-01-04 13:31:00` = **2.017 小时**
4. `2012-12-31 10:15:00 -> 2012-12-31 10:31:00` = **0.267 小时**
5. `2013-01-04 10:15:00 -> 2013-01-04 10:31:00` = **0.267 小时**

### 1.4 对 anchor 附近缺口的量化结论
- 已知 anchor 前最近 `1m`：`2012-12-31 15:00:00+08`
- 已知 anchor 后最近 `1m`：`2013-01-04 09:01:00+08`
- 前后可用最近值总间隔：**约 90.017 小时**

因此，**关键窗口内确实存在跨天/跨多天的大缺口，不是分钟级零散缺口。**

## 四、判断：局部缺失 vs 系统性缺失

### 4.1 可支持的判断
**当前最稳妥结论：`au9999` 存在局部时间段集中缺失，偏向 symbol 级 / 局部窗口问题，而不是已证实的整段历史系统性缺失。**

### 4.2 判断依据
1. **关键窗口内缺口很大且连续**
   - `2012-12-31 15:00 -> 2013-01-04 09:01` 约 `90.017h`
2. **异常集中在单周**
   - `2012-12-31` 当周仅 `450` 条，而前后周都是 `1125` 条
3. **月度覆盖总体连续**
   - `2012-11`、`2012-12`、`2013-01` 均有大体稳定的月度覆盖
4. **异常前后能恢复**
   - `2012-12-28` 前正常，`2013-01-07` 起恢复正常
5. **横向对照不足**
   - 同时期其他 `au*` symbol 几乎无分钟数据，不足以证明是“全局历史库系统性问题”

### 4.3 不建议过度下结论的点
目前证据**不足以**直接下以下更强结论：
- “整个 2012-12 / 2013-01 历史都系统性缺失”
- “所有黄金相关 symbol 在该时段都缺失”
- “一定是 ingestion / 回补流程在该周失败”
